## 2025-05-15 - [Nested Data Access & Duplicate Handling]
**Learning:** The application data in 'data.js' was being flattened from a deeply nested structure on every single lookup call, leading to significant O(N) performance overhead in search and cross-referencing. Furthermore, the presence of duplicate IDs and Names across different sections required the indexing logic to strictly preserve the "first-hit" priority to maintain functional parity with the original '.find()' implementations.
**Action:** Use lazy-initialized memoization and Map-based indexing for static data structures. When implementing indexing, explicitly handle duplicates by checking for existing keys before insertion to guarantee that the first occurrence in the original traversal order remains the primary reference.
